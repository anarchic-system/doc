1. Preamble

Why another document about GnuPG?

Well, if it is just about setting up a key, one can read the (from quite good to absolutely excellent) documents listed in [id_preamble_ressource].

The goal of this document is first to provide a (functional) goal / use case (see [id_preamble_overview]. The remaining of the document tries to be consistent with that goals. This document also tries to always provide a “high level” view because one can quickly get losts in details.

There is a important part about preparing for after the bus came (see [id_share_key]).

Then once one can trust the key, one can build upon it (see [id_usage]).

There are a few other things. And to finish this document is written for GnuPG 2.1.15.

1.3. Goal / use case

Use the GnuPG keys as the most trusted corner stone of as many usages as possible

  1. to encrypt passwords or “confidential” information,

  2. to authenticate against SSH servers,

  3. to sign or encrypt emails,

  4. to open LUKS crypto-containers,

  5. to open remote LUKS crypto-containers,

  6. to sign Git commits,

  7. to authenticate locally,

  8. to encrypt XMPP chats,

  9. to login to a VPN.

by

  1. implementing best practices
    [The “s“ part is the difficult part; one must sometimes make a choice among different possibilities and still have the whole solution working properly.]
    ,

  2. using GnuPGcard to store private keys,

    1. one (certification) GnuPGcard with the master|primary key that

      • is used only within a pinpad reader,

      • is kept safely,

      • is only used for certification (other key signature) or subkey modification purpose,

    2. one (daily) GnuPGcard without the master|primary key that

      • stays at home,

      • is for all daily usages (encryption, signature and authentication),

      • only requires the --import <file_gpg_subkey> (or better the fetch for on-line machines) command as a prerequisite (and the gpg.conf if possible).

    3. another (daily) GnuPGcard without the master|primary key that

      • is always with me in the USB reader, on my (physical) key ring,

      • has same usage as above; for all daily usages (encryption, signature and authentication),

      • only requires the --import <file_gpg_subkey> (or better the fetch for on-line machines) command as a prerequisite (and the gpg.conf if possible).

  3. having

    1. a backup of the private key

      • on a USB stick,

      • on paperkey,

      • shared among trusted friends or family members,

    2. a revocation certificate ready.

  4. publishing (public) key

    1. normaly,

    2. using Tor.

1.4. Process overview

The key setup process is a bit lengthy but rather simple:

  1. environment preparation

    1. setup a clean room environment (see [id_setup_clean_room_environment]),

    2. be sure the physical reader and the smartcard are working fine (see [id_setup_reader_card]),

    3. mount the USB key (see [id_setup_clean_room_environment_usbkey]) to be used as $GNUPGHOME,

    4. check the configuration files gpg.conf and gpg-agent.conf are OK

  2. generate keys

    1. a (primary|master) key (see [id_setup_create_key_master]),

    2. a signature subkey (see [id_setup_create_subkey_signature]),

    3. an encryption subkey (see [id_setup_create_subkey_encryption]),

    4. an authentication subkey (see [id_setup_create_subkey_authentication]),

  3. backup

    1. the (primary|master) key and its subkeys (see [id_backup_key]),

    2. the revocation certificate (see [id_backup_revocation_certificate]),

    3. the key trust (see [id_backup_trust]),

    4. the printable / manual way (see <<id_backup_paperkey),

  4. for each daily GnuPGcard

    1. import the original (master|primary) key (see [id_setup2_gnupgcard_import_masterprimary_private_original])

    2. move subkeys to the daily GnuPGcard (keytocard) (see [id_setup2_gnupgcard_move_subkey_to_daily_gnupgcard] and [id_setup2_gnupgcard_move_subkey_to_daily_gnupgcard_bis])

    3. export subkeys; this step is already done at the backup stage (see [id_backup_subkey_private]),

    4. remove (master|primary) key (see [id_setup2_gnupgcard_remove_masterprimary_private_key]),

    5. import subkeys (that are actually stubs as they have been moved to the GnuPGcard above) (see [id_setup2_gnupgcard_import_subkey]),

    6. check (see [id_final_check_gnupgcard_daily]),

      • result

        • master|primary key is non-existant (neither on the machine nor on the GnuPGcard)

        • subkeys are on the GnuPGcard

        • gpg -K ⇒ sec# and 3(ssb>)

  5. for the certification GnuPGcard (used only in pinpad reader)

    1. import the original (master|primary) key (see [id_setup2_gnupgcard_import_masterprimary_private_original]),

    2. move the (master|primary) key to its GnuPGcard (keytocard) (see [id_setup2_gnupgcard_move_masterprimary_to_gnupgcard]),

    3. export subkeys; this step is already done at the backup stage (see [id_backup_subkey_private]),

    4. delete (master|full) key (see [id_setup2_gnupgcard_remove_masterprimary_private_key]),

    5. import subkeys (see [id_setup2_gnupgcard_import_subkey]),

    6. check (see [id_final_check_gnupgcard_cert]),

      • result

        • master is on the GnuPGcard

        • subkeys are in $GNUPGHOME

        • gpg -K ⇒ sec> and 3(ssb)

Then come the steps of

  1. backing up (see [id_backup]) of the

    1. private keys (see [id_backup_key_private],

    2. private subkeys (see [id_backup_subkey_private]),

    3. private key on paper (see [id_backup_paperkey]),

    4. trust (see [id_backup_trust]),

  2. generating a revocation certificate (see [id_backup_revocation_certificate]),

  3. backing up and making system wide the GnuPG configuration (see [id_]),

  4. preparing for the bus day (see [id_share_key]).

    Be ready

    The bus might be late but not that late that what was encrypted with the GnuPG key could be brute-forced.

  5. and publishing the (public) key

    1. on key servers (see [id_publickey_publication_keyserver]),

    2. during keysigning parties to have it signed (see [id_publickey_publication_signing_party]).

  • sign Git’s commit (see [id_usage_git]),

  • encrypt XMPP / Japper / Gajim communications (see [id_usage_gajim]),

  • authenticate a certificate (see [id_usage_authenticate_certificate]),

  • sign a Debian package (see [id_usage_deb_signature]),

  • encrypt a file (see [id_usage_encrypt_file]).

Some maintenance tasks

  • extend the key validity (see [id_maintenance_extend_key_validity]),

2.4. The pinpad reader case

Dpending on the pinpad used, there might be necessary to use a hack to have it working (working being defined by “enter the PIN on the pinpad and not on the computer keyboard”). Pinpads that can get a variable PIN length do not need the hack.

See http://wiki.gnupg.org/CardReader/GemaltoPC for the hack (and the explanations).

Synthesis

  1. prepare file <file_gpgcard_login> with login information,

  2. edit GnuPGcard and set the login information to the file <file_gpgcard_login>.

Prepare file <file_gpgcard_login> with login information
lerouge$
^TP=8,8
“^T” is a control character. With Vim, enter “ctrl+v” and then “ctrl+t”.
Edit GnuPGcard login information
$ gpg --card-edit
[…]
gpg/card> admin
Admin commands are allowed

gpg/card> login < <file_gpgcard_login>
[…]
gpg/card> quit
$

This has two drawbacks :

  1. the user and admin PINs have the same length,

  2. both are set to eight digits.

And then this is working!

2.5.1. $GNUPGHOME structure

In $GNUPGHOME:

  • pubring.kbx — file storing the keyring,

  • trustdb.gpg — file storing trust for each key,

  • private-keys-v1.d/ — directory where private keys are stored (stup if they are moved to a GnuPGcard),

  • keyservers_certificates — directory with certificate used by key servers to ensure we are talking to legit servers (at least if the certificate stored in this directory was checked properly),

  • configuration files (see above),

  • public key,

  • public authentication subkey in SSH format,

  • information for keysigning parties.

The second form of the command has the special property to render the secret part of the primary key useless; this is a GNU extension to OpenPGP and other implementations can not be expected to successfully import such a key. Configuration — Local machine — $(HOME)/.ssh/config
Host <host_id>
   Hostname <fqdn_remote_machine>
   User <login_on_remote_machine>
   ## forward to remote_machine the local_machine GnuPG agent socket
   RemoteForward $(GNUPGHOME)/S.gpg-agent $(GNUPGHOME)/S.gpg-agent.remote
   ## fail hard
   ExitOnForwardFailure yes
Variable expansion

OpenSSH is not expanding environment variables. It means that path used for the option RemoteForward must be fully qualified (such as /home/your_login/.gnupg/).

Use case

The public key must be imported from a

  • URI (gpg --recv-key <key_id> but this requires dirmngr on the remote machine),

  • local copy (gpg --import <path_key>).

GnuPG directory

The $GNUPGHOME directory should exist before doing the first ssh. Otherwise the connection will failed thanks to the option ExitOnForwardFailure that is enabled.

  1. ssh <fqdn_remote_machine>

  4. enter PIN of the key (in the GnuPGcard plugged the local machine).
